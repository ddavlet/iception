Project Features

Backup Service: Restic

For the backup solution, I chose restic. Restic is a fast, secure, and efficient backup program that offers features like encryption and deduplication. It’s an ideal choice for ensuring that data is safely backed up and can be restored when needed. This choice allows for robust data protection and easy recovery in case of any issues with the primary data.

During the defense, I'll explain how Restic’s capabilities align with the project’s needs and why it was the best fit. The backup process is crucial for ensuring data integrity and reliability, which is why it was a priority in this setup.

Security Measures

Security is a critical aspect of this project. To enhance safety, I implemented the following measures:

  • Secrets Management: I used Docker secrets to manage sensitive information. This practice prevents hard-coding passwords or credentials in the Docker images, which minimizes the risk of exposing confidential data.
  • Safe Entrypoint with Tini: Instead of relying on hacky solutions like `tail -f`, I used Tini as the entrypoint for containers. Tini is a small but powerful init system that handles process reaping and signal forwarding, ensuring a more robust and stable container runtime.
  • No Hardcoded Passwords: I made sure that no passwords or sensitive information are included in the Dockerfiles or other configuration files. This approach is crucial for maintaining security and avoiding potential leaks.
  • Environment Variables: I used environment variables and a `.env` file to manage configuration settings securely. This method keeps sensitive data out of source control and ensures that configurations can be easily adjusted without modifying the code.

Networking Setup

In this project, I configured a Docker network to facilitate communication between containers. This setup is essential for maintaining isolated and secure interactions among services, preventing unauthorized access and ensuring smooth operation.

By avoiding the use of deprecated or insecure networking methods such as `network: host` or `--link`, I adhered to best practices and ensured that the network configuration is both modern and effective.

Docker Setup Details

The Docker setup for this project includes multiple services running in their dedicated containers, each built from custom Dockerfiles. This approach guarantees that each service is tailored to its specific requirements and performs optimally.

Here’s a brief overview of the services included:

  • NGINX: Configured with TLSv1.2 or TLSv1.3 only, serving as the single entry point to the infrastructure.
  • WordPress + php-fpm: A container running WordPress with php-fpm, optimized for performance and security.
  • MariaDB: A container dedicated to running MariaDB, ensuring data management is handled separately from web services.
  • Volumes: Two volumes are used – one for the WordPress database and another for website files, providing persistent storage and data isolation.

Bonus Part

In addition to the mandatory setup, I implemented several bonus features to enhance the project further:

  • Redis Cache: Set up a Redis cache to improve the performance and speed of the WordPress website by managing the cache more efficiently.
  • FTP Server: Configured an FTP server container that connects to the volume of the WordPress website. This setup facilitates file transfers and management of website files.
  • Static Website: Created a simple static website as an example of an additional service. This website is a showcase for presenting resumes or personal projects and is implemented using HTML and CSS.
  • Adminer: Deployed Adminer, a lightweight database management tool, to manage MariaDB databases with ease.
  • Additional Useful Service: Added restic app to back up all important data

Each bonus feature was carefully selected to add value to the project, demonstrating not only technical skills but also creativity and practical problem-solving.

Project Structure

The project is organized with a clear and logical directory structure to facilitate development and management:

  • Makefile: Located at the root of the directory, it orchestrates the build process for Docker images and services.
  • srcs/: Contains all the source files for Docker configurations and service definitions.
  • srcs/requirements: Includes specific directories for each required service, such as NGINX, MariaDB, and WordPress, with their respective Dockerfiles and configurations.
  • secrets/: Stores sensitive information like passwords and API keys securely, managed through Docker secrets.
  • .env: Contains environment variables used for configuring services.

This structure ensures that the project is modular, easy to navigate, and maintainable. Each component is clearly defined and organized, supporting efficient development and deployment.
